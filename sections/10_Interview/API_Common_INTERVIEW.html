Introduction

Functional testing of APIs ensures that each endpoint behaves according to its specifications. It validates input/output, workflows, and error handling. Unlike UI testing, it directly tests the business logic via requests and responses.


Fundamentals
  • What is an API and why is it important in software testing?

    An API (Application Programming Interface) allows communication between software components. In testing, APIs are crucial for validating backend logic, data exchange, and integration points without relying on UI.

  • What are the key differences between REST and SOAP APIs?

    REST is lightweight, uses HTTP, and supports multiple formats (JSON, XML). SOAP is protocol-based, XML-only, and includes strict standards like WS-Security and WSDL.

  • What is the structure of an HTTP request and response?

    Request: Method, URL, Headers, Body. Response: Status Code, Headers, Body. Both are used to exchange data between client and server.

  • What are common HTTP methods used in API testing?

    GET (retrieve), POST (create), PUT (replace), PATCH (update), DELETE (remove).

  • What is the difference between PUT and PATCH?

    PUT replaces the entire resource, while PATCH updates only specified fields.

Tools & Frameworks
  • What is Postman and how is it used for API testing?

    Postman is a GUI-based tool for sending HTTP requests, validating responses, writing test scripts, and organizing collections for automation.

  • How does Newman complement Postman in CI/CD?

    Newman is Postman's CLI runner. It executes collections in CI pipelines, supports HTML/JSON reports, and integrates with Jenkins, GitHub Actions, etc.

  • What are some alternatives to Postman?

    REST Assured (Java), Karate (DSL), Cypress (JavaScript), Insomnia (GUI), and HTTPie (CLI).

  • How do you automate API tests using CI platforms?

    Use CLI runners like Newman, integrate with CI tools (Jenkins, GitHub Actions), inject secrets, and generate reports as artifacts.

Security & Validation
  • How do you test for authentication and authorization?

    Use valid/invalid tokens, test role-based access, and validate response codes (401, 403). Include header injection and token expiry scenarios.

  • What is token-based authentication?

    Clients receive a token (e.g., JWT) after login, which is sent in headers for subsequent requests. Tokens are validated server-side.

  • How do you handle secure API key injection in CI/CD?

    Store keys in secrets vaults or CI credential stores. Inject dynamically using environment overrides (e.g., jq in GitHub Actions).

  • What are common API security vulnerabilities?

    Injection attacks, broken authentication, excessive data exposure, insecure endpoints, and lack of rate limiting.

Troubleshooting & Debugging
  • How do you debug a failing API test?

    Inspect request/response logs, validate headers and payloads, check environment variables, and isolate test cases.

  • What tools help inspect payloads?

    Postman console, browser dev tools, curl, HTTP proxies (Fiddler, Charles), and CLI tools like HTTPie.

  • How do you handle flaky API failures?

    Add retries, increase timeouts, validate backend stability, and isolate external dependencies.

  • How do you validate performance?

    Use response time assertions, load testing tools (JMeter, k6), and monitor latency across environments.

Advanced Concepts
  • What is contract testing?

    Validates that API consumers and providers agree on request/response formats. Tools: Pact, Spring Cloud Contract.

  • How do you test APIs in microservices?

    Use service mocks, test orchestration flows, validate inter-service communication, and isolate dependencies.

  • What is mocking and stubbing?

    Mocking simulates external services; stubbing returns predefined responses. Useful for testing without real dependencies.

  • How do you handle dynamic data?

    Use environment variables, pre-request scripts, and random data generators (e.g., Faker) to simulate realistic inputs.

Important Tips for Interview Candidates
  • Showcase reusable test scripts and assertions

    Interviewers value candidates who write clean, maintainable tests. Mention how you use Postman's pm.test() and pm.expect() to validate responses consistently.

    Example: “In my Postman collection, I created a shared test script that checks for status code, response time, and JSON schema. This helped reduce duplication across 50+ requests.”

    pm.test("Valid response", () => {
  pm.response.to.have.status(200);
  pm.expect(pm.response.responseTime).to.be.below(500);
});
  • Demonstrate modular environments and secure secrets handling

    Talk about how you manage different environments (dev, QA, prod) and inject secrets securely in CI/CD pipelines.

    Example: “I used Postman environments with variables like {{base_url}} and {{auth_token}}, and injected secrets via GitHub Actions using secrets.POSTMAN_API_KEY.”

    env:
  POSTMAN_API_KEY: ${{ secrets.POSTMAN_API_KEY }}
  ENV_OVERRIDES: $(jq -r '.values[] | "\(.key)=\(.value)"' overrides.json)

    Highlight fallback logic or error handling for missing secrets to show robustness.

  • Explain real-world CI/CD automation experience

    Interviewers appreciate candidates who’ve implemented automation beyond local testing. Share how you integrated API tests into pipelines.

    Example: “I configured Newman in GitHub Actions to run Postman collections, generate HTML reports, and send Slack alerts on failure. This gave our team instant feedback.”

    - name: Run API Tests
  run: |
    newman run collection.json \
      -e environment.json \
      -r cli,html \
      --reporter-html-export results.html

- name: Notify Slack
  if: failure()
  run: |
    curl -X POST -H 'Content-type: application/json' \
    --data '{"text":"❌ API tests failed in CI"}' ${{ secrets.SLACK_WEBHOOK }}

    Bonus: Mention how this improved test coverage or reduced manual effort.

🧪 API Testing - Behavioral Interview Questions

1. Tell me about a time you found a critical bug during API testing.

Answer: While testing a payment API, I noticed that the endpoint accepted invalid card expiry dates. I created a test case with a past expiry and confirmed it processed successfully. I flagged it, collaborated with devs, and helped implement stricter validation. This prevented potential financial discrepancies.

2. How do you handle flaky API tests in CI/CD pipelines?

Answer: I first isolate the flaky test by running it independently. Then I check for external dependencies like rate limits or unstable environments. I add retry logic using newman-reporter or wrap assertions with conditional checks. I also log response times and status codes to identify patterns.

3. Describe a situation where you had to test an API without documentation.

Answer: In one project, the API lacked documentation. I used Postman’s proxy and browser dev tools to capture requests. Then I reverse-engineered the endpoints and created a working collection. I documented everything in markdown and shared it with the team to accelerate onboarding.

4. How do you ensure secure handling of API keys in your test automation?

Answer: I avoid hardcoding secrets. Instead, I use environment variables injected via CI/CD tools like GitHub Secrets or GitLab CI. For local runs, I use a Vault-backed script to override Postman environments dynamically using jq. This ensures secrets are never exposed in logs or version control.

5. How do you mentor junior testers in API testing?

Answer: I start with fundamentals—HTTP methods, status codes, and JSON structure. Then I walk them through Postman collections, reusable tests, and environment setup. I use scenario-based guides with collapsible sections and icons to make learning modular and accessible. I also encourage them to write their own test cases and review them together.
