Performance Testing of APIs

What is Performance Testing of APIs?

API performance testing evaluates how reliably and quickly APIs respond under different levels of load. It focuses on response times, throughput, error rates, and stability over time—ensuring your services meet SLAs/SLOs and provide a smooth user experience.

Latency (p50/p95/p99) Throughput (RPS) Error Rate CPU / Memory Scalability Saturation

Core Concepts to Master
  • Latency percentiles (p50/p95/p99)
  • Throughput (requests/second)
  • Concurrency vs RPS
  • Load models: baseline, load, stress, spike, soak
  • Bottlenecks: app, DB, network, external deps
  • Warm-up effects & caching behavior
  • Timeouts / retries / circuit breakers
  • SLIs/SLOs/SLAs
  • Data seeding & realistic payloads
  • Monitoring: logs, metrics, traces
Tip: Always isolate variables—test one change at a time (e.g., DB index, cache toggle, instance size).

What to Validate (How to Test Performance)
  • Response time targets (avg, p95, p99)
  • Throughput targets (e.g., 500 RPS sustained)
  • Error budgets (e.g., <0.1% 5xx)
  • Scalability under concurrency ramps
  • Stability under soak (2–8 hours)
  • Resiliency tests: timeouts, retries, chaos/fault injection

Recommended Test Types

  • Baseline (sanity), Load (steady), Stress (limit), Spike (burst), Soak (endurance)

Workflow

  1. Define SLIs/SLOs & success criteria
  2. Pick target endpoints & payloads (auth, headers, caching)
  3. Prepare realistic data (seed DB, unique IDs)
  4. Model traffic (RPS/concurrency, think-time)
  5. Run tests, monitor infra/app + collect percentiles
  6. Analyze bottlenecks, iterate, and re-test

Test Scenarios & Test Data (Examples)

Scenarios

  • GET /products at 300 RPS steady (cache warm vs cold)
  • POST /orders at 50 RPS with unique payloads
  • Spike 10→500 RPS for 60s on /search
  • Soak test 2h on checkout flow (auth → cart → order)
  • Concurrent reads/writes on inventory endpoints

Assignments (Hands-on)
  1. Design SLOs for a sample catalog API (p95 < 300ms @ 200 RPS, errors <0.1%).
  2. Write a k6 script to load test /products at 200 RPS for 5 minutes.
  3. Compare cold vs warm cache runs and document deltas.
  4. Run a spike test (10→500 RPS→10) and capture error spikes + saturation point.
  5. Create performance assertions (e.g., p95 < 400ms) that fail the CI job if breached.

Minimal k6 Example

// save as perf_products.js import http from 'k6/http'; import { check, sleep } from 'k6'; export const options = { stages: [ { duration: '30s', target: 50 }, { duration: '2m', target: 200 }, { duration: '30s', target: 0 }, ], thresholds: { http_req_duration: ['p(95)<400'], http_req_failed: ['rate<0.01'], }, }; export default function () { const res=http.get('https://api.example.com/products'); check(res, { 'status is 200' : (r)=> r.status === 200, }); sleep(0.2); }

CI/CD Integration

  • Run smoke perf test on every PR
  • Nightly soak test with trend reports
  • Fail pipeline if thresholds breach (e.g., p95 or error rate)

Tips & Best Practices
  • Test against prod-like infra & data volumes.
  • Warm up services to avoid cold-start bias in percentiles.
  • Keep payloads & headers realistic (auth, content-type, caching).
  • Separate read vs write profiles.
  • Measure server & DB metrics alongside client timings.
  • Use percentiles (p95/p99), not only averages.
  • Automate thresholds; make regressions visible in CI.
