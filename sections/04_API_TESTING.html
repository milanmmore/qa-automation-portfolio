Introduction

API testing focuses on verifying that application programming interfaces (APIs) behave as expected. It ensures system reliability, detects issues early in the cycle, and improves test coverage beyond UI.


Fundamentals
  • ‚öôÔ∏è Functional API Testing

    Ensures each endpoint behaves according to requirements by validating requests, responses, status codes, and workflows.

    • ‚úÖ Correct status codes (200, 400, 401, 500, etc.)
    • ‚úÖ Input/output validation
    • ‚úÖ CRUD operations (Create, Read, Update, Delete)
    • ‚úÖ Business rule validation
  • üìä Non-Functional API Testing

    Focuses on how well the API performs under different conditions ‚Äî speed, security, and reliability.

    • ‚ö° Performance ‚Üí Response time, throughput, scalability
    • üîí Security ‚Üí Authentication, authorization, data protection
    • üîÑ Reliability & availability under load
    • üåç Compatibility across environments & versions

    • üëâ In short:
    Functional = Are we building the right thing?
    Non-Functional = Is it fast, secure, and reliable enough?

Key Concepts

üîó 1. Endpoints

The URL paths where APIs are accessed.

Example:

GET https://api.example.com/users/123

Tip: Always document endpoints with required parameters and expected responses.

üì§ 2. HTTP Methods

Define the type of operation the API performs:

  • GET: Retrieve data
  • POST: Create new data
  • PUT: Update existing data
  • DELETE: Remove data

Tip: Ensure the API adheres to RESTful conventions.

üì¶ 3. Request Payloads & Parameters
  • Query Parameters: Appended to the URL 
    ?userId=123
  • Body Parameters: Sent in the request body (e.g., JSON for POST)

Tip: Validate edge cases like missing, null, or invalid values.

üì• 4. Responses & Status Codes

APIs return structured data and HTTP status codes like:

  • 200 OK
    : Success 
  • 201 Created
    : Resource created 
  • 400 Bad Request
    : Client error 
  • 401 Unauthorized
    : Auth failure 
  • 500 Internal Server Error
    : Server crash

Tip: Test all possible response scenarios, including error handling.

üßæ 5. Headers

Carry metadata like content type and authentication tokens.

Example:

Authorization: Bearer <token>

Tip: Validate headers for correctness and completeness.

‚úÖ 6. Assertions

Validate API behavior by checking:

  • Status codes
  • Response body content
  • Response time

Tip: Use tools like Postman, REST Assured, or JSON Schema Validator for automated assertions.

üîê 7. Authentication & Authorization

APIs often require secure access:

  • API Keys
  • OAuth2 Tokens
  • JWTs

Tip: Test role-based access and token expiry scenarios.

üìä 8. Performance & Reliability

Measure response time, throughput, and stability under load.

Example:

GET /weather

Ensure response within 200ms under 100 concurrent users.

Tip: Use JMeter or Gatling for load and stress testing.

  • ‚ñ∂ Best Practices
    • 1. ‚úÖ Validate All HTTP Methods
      Test GET, POST, PUT, DELETE, PATCH thoroughly.
      Ensure each method behaves as expected and follows RESTful conventions.
    • 2. üîê Security Testing
      Check for unauthorized access, token expiry, and role-based permissions.
      Validate headers like Authorization, Content-Type, and CORS policies.
      Test for vulnerabilities: SQL injection, XSS, and rate limiting abuse.
    • 3. üìä Data Validation
      Test with valid, invalid, missing, and boundary inputs.
      Confirm that error messages are meaningful and status codes are accurate (e.g., 400, 401, 403, 404, 500).
    • 4. üìà Performance & Load Testing
      Simulate high traffic to test response time, throughput, and SLA adherence.
      Use tools like JMeter, Postman Runner, or Newman for automation.
    • 5. üîÅ Regression Testing
      Re-run tests after every deployment or code change to ensure nothing breaks.
      Automate critical test cases to catch regressions early.
    • 6. üìÑ Contract & Schema Validation
      Use tools like Swagger, OpenAPI, or Postman schema validation to ensure responses match expected formats.
    • 7. üß™ Functional Testing
      Validate business logic: e.g., ‚Äúuser exists‚Äù ‚Üí 200 OK, ‚Äúinvalid token‚Äù ‚Üí 401 Unauthorized. Test workflows across endpoints (e.g., login ‚Üí fetch profile ‚Üí update settings).
    • 8. üßº Environment Isolation
      Use mock servers or sandbox environments to avoid polluting production data.
      Isolate test data and reset states between runs.
    • 9. üìö Documentation Testing
      Ensure the API behaves exactly as described in its documentation. Flag discrepancies between actual behavior and documented specs.
    • 10. üß† Use Meaningful Test Case Templates
      Create templates that include all relevant information:
      - Test Case ID
      - Title
      - Method
      - Endpoint
      - Input
      - Expected Output
    • 11. Example Test Case
                                  Test Case ID: API_TC_001
                            Title: Verify login with valid credentials
                            Method: POST
                            Endpoint: /api/login
                            Input: { "email": "valid@example.com", "password": "correct123" }
                            Expected: 200 OK, message: "User exists!"

      Use environment variables, validate status codes, automate with Postman/Newman, etc.
